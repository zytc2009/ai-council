### 整体评价

这个问题本身是有价值的，但如果把“model”和“harness”对立起来，往往会把讨论带偏。我的判断是：在能力上限上，模型更重要；在实际产出和稳定性上，harness 往往同样重要，甚至在工程环境里更决定结果。一个强模型放进糟糕的 harness 里，会持续浪费能力；一个一般模型配上优秀的 harness，能显著逼近更强模型的实际效果。真正该问的不是“哪个更重要”，而是“当前瓶颈到底在能力上限，还是在能力释放”。

### 关键要点

1. 模型决定了系统的理论上限。推理深度、知识迁移、代码生成质量、长程一致性，这些核心能力主要由模型本身决定，harness 很难凭空补出不存在的能力。
2. Harness 决定了模型能力能否稳定落地。任务分解、工具调用、上下文管理、检索质量、失败恢复、评测反馈，这些工程机制直接影响同一个模型最终能产出多少有效价值。
3. 在真实系统里，瓶颈常常先出现在 harness，而不是模型。很多团队在模型还没被充分用好的情况下就急着换更强模型，结果成本上升了，但效果提升有限，因为问题其实出在提示链路、状态管理和评测闭环。

### 补充建议

建议把这个问题改写成一个更可操作的判断框架：  
先问“当前失败案例里，多少是模型不会，多少是系统没把问题交给模型处理好”。具体做法是：

1. 建一组失败样本，按原因标注为“能力不足”或“编排不足”。
2. 做二维对比实验：固定 harness 换模型，固定模型换 harness。
3. 看边际收益而不是看绝对表现，判断下一步该投模型升级还是系统优化。
4. 优先补最短板：如果输出不稳定、上下文丢失、工具乱用，先修 harness；如果已经有良好编排但核心推理仍然不够，再换更强模型。

如果必须给一句结论，我会说：模型决定你能到多高，harness 决定你能不能真的到那里。
