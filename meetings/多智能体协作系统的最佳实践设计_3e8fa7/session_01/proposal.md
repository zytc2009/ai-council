# 方案：多智能体协作系统的最佳实践设计

> 状态：草案
> 版本：1
> 日期：2026-04-02

---

## 1. 问题描述

随着 LLM 应用从单一智能体扩展到多智能体协作系统，工程团队面临以下核心挑战：

1. **协作复杂度失控**：多智能体间的通信协议、任务路由、上下文共享缺乏统一标准，导致集成错误率高、调试困难。
2. **可靠性保障缺失**：LLM 调用具有高延迟、不可预测性和输出随机性，传统的同步调用和幂等性假设在此场景下不再成立。
3. **架构演进代价高**：过度提前引入分层架构、熔断器、背压机制等复杂模式，导致配置爆炸和维护成本超过收益；但缺乏规划又导致后期重构成本居高不下。
4. **上下文管理混乱**：共享 Context 对象在无写权限控制的情况下容易退化为隐式全局状态，智能体迭代时 schema 不兼容问题被忽视。

本方案旨在提供一套**渐进式、可验证、低维护成本**的多智能体协作系统设计最佳实践。

---

## 2. 方案概述

本方案的核心思路是**按问题驱动复杂度，而非按理想架构驱动实现**。从最小可行的线性 Pipeline 出发，以可观测性和接口契约为基础能力，在实测出现瓶颈时再按需引入并行化、熔断、动态路由等高级机制。这一策略避免了"理想系统"带来的配置爆炸，同时保留了架构演进的空间。

在接口与数据契约层面，方案强制要求每个智能体声明标准化的输入/输出 Schema，并通过 `ReadContext`/`WriteContext` 分离模式从编译期杜绝意外的全局状态写入。针对 LLM 输出的随机性，对关键幂等决策引入 `temperature=0` + 结构化 JSON 输出作为确定性锚点，替代不可靠的通用重试机制。

在成本与性能层面，方案推荐高频低复杂度任务优先分配给 Haiku 4.5（成本约为 Sonnet 的 1/3），并在 CI 流水线中集成每个智能体的 P50/P99 延迟与 token 消耗基准测试，将性能退化问题拦截在合并前，而非集成后才发现。

## 3. 技术方案

### 3.1 架构设计

采用**声明式 DAG + 两阶段演进**的架构策略：

**阶段一：线性 Pipeline（初始态）**

```
Input → Agent A → Agent B → Agent C → Output
```

- Orchestrator 解析 YAML/TOML 声明的有向无环图（DAG），驱动线性执行
- 调度策略变更仅修改配置文件，不触碰 Orchestrator 核心代码
- 所有通信采用异步消息队列（如 Redis Stream / 内存队列），避免同步阻塞

**阶段二：按需扩展（瓶颈驱动）**

| 触发条件 | 引入机制 |
|---------|---------|
| 实测出现串行瓶颈 | 并行子图（DAG 分支） |
| 智能体连续失败超阈值 | 熔断器 + 降级路由 |
| 队列溢出 | 背压机制 |
| 智能体数量 > 10 | 动态能力注册表 |

**动态注册表引入门槛**：智能体数量 ≤ 10 时，优先使用静态 YAML 声明式路由；超过 10 个时，智能体启动时声明能力标签，Orchestrator 动态匹配路由。

### 3.2 核心模块

**1. Schema 契约层（强制）**

每个智能体必须声明：
- `InputSchema`：Pydantic 模型或 JSON Schema，入口强制校验
- `OutputSchema`：结构化输出格式，支持版本号字段（`schema_version`）
- `ErrorFormat`：标准化错误结构 `{code, message, retryable}`

**2. Context 管理模块**

```python
@dataclass(frozen=True)
class ReadContext:
    """所有智能体可读，不可修改"""
    task_id: str
    global_config: dict
    shared_state: dict  # 仅快照，写入需通过 delta

@dataclass
class ContextDelta:
    """智能体显式输出的状态变更，由 Orchestrator 合并"""
    updated_fields: dict
    agent_id: str
    timestamp: float
```

- `ReadContext` 为不可变对象（`frozen=True`），类型系统阻止意外写入
- 智能体通过返回 `ContextDelta` 声明状态变更，Orchestrator 统一合并
- Context 对象携带 `schema_version` 字段，支持向后兼容字段扩展

**3. 确定性输出模块（幂等关键路径）**

对任务分配、优先级排序等需要幂等性的决策：
- Prompt 要求输出结构化 JSON（禁止自然语言回答）
- 调用参数设置 `temperature=0`
- 对输出结果做 Schema 校验，校验失败触发结构化重试（非盲目重试）

**4. 可观测性模块（强制）**

每个智能体输出标准化结构化日志：

```json
{
  "agent_id": "summarizer-v2",
  "task_id": "abc-123",
  "latency_ms": 1240,
  "tokens_input": 512,
  "tokens_output": 128,
  "status": "success",
  "timestamp": "2026-04-02T08:38:00Z"
}
```

Orchestrator 实时消费日志，驱动熔断判断与告警。

**5. 模型选择策略**

| 任务类型 | 推荐模型 | 理由 |
|---------|---------|------|
| 结构化输出、Schema 校验、代码格式化 | Haiku 4.5 | 高频低复杂度，成本约为 Sonnet 的 1/3 |
| 主要开发工作、Orchestrator 调度逻辑 | Sonnet 4.6 | 最佳编码能力 |
| 复杂架构决策、深度推理任务 | Opus 4.6 | 最强推理能力 |

### 3.3 数据流

## 4. 实施计划

### 4.1 阶段划分

**Phase 0：基础设施（前置条件）**
- 定义全局通信 JSON Schema（输入/输出/错误格式）
- 实现 `ReadContext` + `ContextDelta` 模式
- 搭建结构化日志管道

**Phase 1：线性 Pipeline MVP**
- 实现 YAML DAG 解析器与线性执行引擎
- 每个智能体接入 Schema 校验（Pydantic）
- 沙箱测试环境搭建，Mock 依赖，覆盖率目标 ≥ 80%

**Phase 2：可靠性加固（按瓶颈驱动）**
- CI 集成性能基准测试（P50/P99 延迟 + token 消耗）
- 对幂等关键路径引入 `temperature=0` + 结构化 JSON 输出
- 根据实测数据决定是否引入熔断器

**Phase 3：规模化扩展（>10 智能体时）**
- 引入动态能力注册表
- DAG 支持并行分支
- 按需引入背压机制

### 4.2 优先级

| 优先级 | 项目 | 理由 |
|--------|------|------|
| P0（必须） | 标准化接口 Schema | 集成错误率的核心杠杆 |
| P0（必须） | 结构化日志与可观测性 | 实践 ROI 最高，是所有后续优化的基础 |
| P0（必须） | ReadContext / ContextDelta 分离 | 防止全局状态污染，越晚引入改造成本越高 |
| P1（应该） | YAML DAG 声明式路由 | 降低 Orchestrator 耦合度 |
| P1（应该） | 幂等决策的确定性锚点 | 关键路径可靠性保障 |
| P1（应该） | CI 性能基准测试 | 防止性能退化进入主干 |
| P2（可选） | 熔断器 / 背压机制 | 按实测瓶颈引入，非提前投入 |
| P2（可选） | 动态能力注册表 | 仅在智能体数量 > 10 时引入 |

## 5. 风险与应对

| 风险 | 影响 | 应对措施 | 提出者 |
|------|------|---------|--------|
| 分层架构过早引入导致配置爆炸 | 高：维护成本超过收益，调试困难 | 采用两阶段策略，按问题驱动引入复杂机制 | claude-sonnet（第2轮） |
| Orchestrator 中心化成为单点瓶颈 | 高：调度策略变更需修改核心组件，违背局部变更原则 | 声明式 DAG 将调度逻辑外移到配置文件，Orchestrator 仅负责解析执行 | claude-sonnet（第2轮） |
| LLM 输出随机性导致幂等性失效 | 高：同一输入产生不同输出，重试机制失效 | 对幂等关键路径设置 `temperature=0` + 结构化 JSON 输出作为确定性锚点 | claude-sonnet（第2轮） |
| Context 对象退化为隐式全局状态 | 中：共享写入导致不可预期的状态污染，调试极难 | ReadContext（不可变）+ ContextDelta（显式变更）+ Orchestrator 统一合并 | claude-sonnet（第2轮） |
| Context Schema 版本不兼容 | 中：智能体迭代时旧版 Context 字段被读取或覆盖 | Context 对象携带 `schema_version`，设计向后兼容字段，提前制定演进策略 | claude-sonnet（第2轮） |
| 过严的 JSON Schema 阻碍快速扩展 | 中：严格校验导致智能体能力迭代受阻 | Schema 设计保留 `extensions` 扩展字段，核心字段严格校验，扩展字段宽松校验 | claude-sonnet（第2轮） |
| 异步通信引入消息乱序或丢失 | 中：任务执行顺序不确定，结果不一致 | 消息携带序列号与幂等 ID，DAG 执行引擎做依赖等待而非盲目消费 | claude-sonnet（第1轮） |

## 6. 未解决的分歧

以下分歧点在本轮讨论中未达成完全共识，建议用户根据具体项目规模和团队情况裁决：

**分歧 1：动态能力注册表的引入门槛**

- **立场 A（第1轮）**：推荐从初始阶段即采用动态注册表，智能体启动时声明能力标签，Orchestrator 动态路由，灵活性高。
- **立场 B（第2轮）**：反对早期引入，动态路由带来运行时不确定性，调试和审计成本高。建议以智能体数量 **>10** 作为引入门槛，10 个以内优先使用静态 YAML 声明。
- **建议裁决方向**：如果系统预期在 6 个月内智能体数量将超过 10，可提前引入注册表架构；否则优先静态路由。

**分歧 2：三层架构（Orchestrator → Coordinator → Worker）的引入时机**

- **立场 A（第1轮）**：推荐从三层结构起步，清晰分离调度逻辑与执行逻辑，便于管理复杂性。
- **立场 B（第2轮）**：反对一步到位，三层 + 熔断 + 背压叠加会产生配置爆炸，建议在 Pipeline 稳定且出现明确的 Coordinator 职责需求后再引入中间层。
- **建议裁决方向**：单团队维护、智能体数量 <15 的项目建议两层（Orchestrator + Worker）起步；跨团队、多业务域的大型项目可考虑三层结构。

## 7. 参会者评估

**claude-sonnet（综合两轮立场）**

对本方案的最终立场：**支持，附条件**

- **强烈支持**的部分：单一职责原则、可观测性优先、异步通信架构、标准化接口契约——这四项是实践中 ROI 最高的设计原则，无论系统规模均应优先落地。
- **支持但需谨慎**的部分：声明式 DAG 和 ReadContext/ContextDelta 模式方向正确，但实现细节（Schema 版本化、Context 合并策略）需在 Phase 0 设计阶段仔细推敲，避免早期决策成为后期债务。
- **保留意见**的部分：熔断器、背压机制、动态注册表的引入时机建议严格遵守"瓶颈驱动"原则，任何超前引入都会在维护成本上产生负向收益。整体方案的价值在于其**渐进式演进路径**，而非最终的完整架构形态。
