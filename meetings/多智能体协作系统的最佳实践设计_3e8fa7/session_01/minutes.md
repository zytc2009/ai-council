## 讨论摘要

### 共识点

1. **单一职责原则**是智能体设计的核心基础，专注单一任务的智能体更易测试、替换和扩展。
2. **可观测性优先**：标准化的日志、指标和追踪输出是系统健康的必要保障，实践 ROI 最高。
3. **异步通信优于同步阻塞**：LLM 调用延迟高且不可预测，消息队列/事件驱动架构是必要选择。
4. **为失败设计**：局部失败不应导致全局崩溃，降级策略是生产级系统的必备能力。
5. **幂等性设计**可减少故障恢复成本，但在 LLM 场景下需在应用层额外处理。

**claude-sonnet（第 1 轮）的最有价值论据：**
- 通信协议标准化（统一 JSON Schema）是协作效率的核心杠杆，直接降低集成错误率。

- 过度中央化的 Orchestrator 本身会成为单点瓶颈——调度策略每次变更都需触碰核心组件，违背局部变更局部影响原则。
- LLM 输出的随机性使幂等性在传统重试机制下无法保证，需通过 `temperature=0` + 结构化 JSON 输出在应用层建立确定性锚点。
- Context 对象的版本化问题在智能体迭代中容易被忽视，需提前设计向后兼容策略。

## 下一步建议

### 架构设计

- [ ] 采用**两阶段启动策略**：先实现线性 Pipeline（A→B→C）验证业务逻辑，再按实测瓶颈引入并行化和熔断机制
- [ ] 用**声明式 DAG**（YAML/TOML）替代命令式路由，Orchestrator 解析执行，调度策略变更仅修改配置文件
- [ ] 仅在智能体数量 >10 时引入**动态能力注册表**，否则优先使用静态路由配置

### 接口与数据契约

- [ ] 为每个智能体定义标准化的**输入 schema、输出 schema 和错误格式**（Pydantic 模型或 JSON Schema 强制校验）
- [ ] 引入 `ReadContext`（全局可读）和 `WriteContext`（仅授权智能体可写）的**写权限分离模式**，通过类型系统在编译期阻止意外写入
- [ ] 为 Context 对象设计**版本化策略**（向后兼容字段或版本号字段），应对智能体迭代时的 schema 不兼容问题

- [ ] 对关键幂等决策（任务分配、优先级排序），在 prompt 中要求结构化 JSON 输出并设置 `temperature=0` 作为**确定性锚点**
