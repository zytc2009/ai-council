# 让多个 AI 帮你把需求问清楚——cli-assistant 需求讨论模式

> 你有没有遇到过这种情况：脑子里有个想法，跟 AI 说了一句，它直接开始写代码，写完你一看——方向完全不对？

问题不在 AI，在于需求本身没说清楚。"做个拼图游戏"这句话，可以实现成网页，也可以是桌面应用；可以拖拽交换，也可以滑动移位；可以有计时，也可以没有。每一个分叉，都可能让开发方向跑偏。

cli-assistant 的**需求讨论模式**，就是在动手之前，先把这些问题问清楚。

---

## 核心思路：让多个 AI 帮你挑漏洞

单个 AI 回答问题，容易顺着你的意思走，把你没说清楚的地方自动脑补掉。多个 AI 同时看一份需求，角度不同，关注点不同，会暴露出截然不同的盲区。

需求讨论模式做的事情，就是把多个 AI CLI（Claude、Codex、Kimi 等）组织成一个协作小组，让它们帮你把需求里的模糊点一一挖出来，澄清完之后，输出一份结构化的需求文档——这份文档可以直接喂给 auto-dev 来驱动自动开发。

---

## 整个流程长什么样

```
你描述需求
    ↓
Phase 1：各 AI 独立分析（互不可见）
    → 每个 AI 输出：已明确的字段 / 待澄清的问题 / 假设前提
    ↓
你回答问题（可以跳过——你也有盲区，没关系）
    ↓
需求理解确认清单
    → AI 综合初步分析，列出：当前理解 / 准备做的假设 / 仍有疑问
    → 你确认，或者纠正误解
    ↓
Phase 2：迭代澄清（无轮数限制）
    → 所有 AI 平等参与，每轮更新对 5 个字段的认知
    → 每轮结束展示仍待澄清的问题，你可以补充
    → 你不回答的问题，AI 给出合理假设继续推进
    → 5 个字段全部收敛，或你输入 d 结束
    ↓
Phase 3：综合输出 requirement.md
    → 结构化需求文档，可直接喂给 auto-dev
```

## 一个真实的例子

用"做个拼图小游戏"这句话启动需求讨论，Phase 1 结束后，Claude 列出了这些问题：

```
[Scope] 碎片移动方式是拖拽到目标位置，还是「滑动拼图」（只有一个空位靠移位）？
[Inputs] 使用内置预设图片，还是支持玩家自己上传？
[Outputs] 在浏览器里玩，还是桌面应用？
[Acceptance] 拼完后只弹「完成」，还是需要计时、计步？
```

用户回答了其中几个，有几个说"不确定"。AI 没有继续追问，而是给出了明确的假设："如果平台未确认，默认为网页（浏览器）实现"，然后基于这些假设继续推进。

最终输出的 `requirement.md`：

```markdown
# Requirement: 做个拼图小游戏

## Goal
开发一款单机拼图小游戏：玩家上传本地图片，系统将其切割成网格碎片，
玩家通过拖拽将碎片还原成原图，完成后记录成绩并展示排行榜。

## Scope
- In scope:
  - 玩家上传本地图片作为拼图素材
  - 支持 3×3 / 4×4 / 5×5 / 6×6 四档难度
  - 拖拽交互：玩家自由拖拽碎片到目标格位
  - 计时功能：游戏开始计时，拼完停止
  - 排行榜：本地存储历史成绩，按用时排序
- Out of scope:
  - 撤销/悔步功能
  - 联网排行榜/多人对战

## Inputs
- 玩家从本地文件系统上传的图片文件

## Outputs
- 游戏界面：打乱后的碎片网格及实时计时器
- 结果页：拼图完成后展示用时
- 排行榜页：历史完成记录

## Acceptance Criteria
- 玩家上传图片后可选择网格规格开始游戏
- 碎片以随机顺序分布，且该分布保证存在完整解法
- 所有碎片归位后判定完成，计时停止
- 成绩写入本地排行榜

## Open Questions
- 运行平台未经用户明确确认：网页还是桌面窗口应用？
- 支持的图片格式与单文件大小上限未指定
```

这份文档有明确的 Goal、清晰的 Scope 边界、可验证的 Acceptance Criteria，以及诚实记录的 Open Questions——没有被强行填满，也没有发明用户没说的东西。

## 对接 auto-dev：需求文档到自动开发的管道

需求讨论结束后，`requirement.md` 自动保存在 `meetings/<议题>/` 目录下。这份文档的格式与 auto-dev 的输入规范完全对齐：

```
cli-assistant 需求讨论
    → meetings/<议题>/requirement.md
        ↓
    auto-dev（读取 requirement.md）
        → 自动生成实现方案
        → 编写代码
        → 运行测试
```

你不需要手动整理需求，也不需要从对话记录里提炼——讨论结束即交付，直接进入下一步。

---

## 几个关键设计决策

**为什么 Phase 2 没有主持人？**

传统的"主持人引导讨论"模式适合观点碰撞，但在需求澄清场景下，主持人会引入不必要的层级感。需求澄清更像是多个分析师各自独立检查同一份文档，不需要有人"引导"——每个 AI 直接盯着 5 个字段看，缺什么填什么。

**为什么不限制轮次？**

需求复杂度不一样，有的两轮就清楚了，有的需要五六轮。固定轮次要么浪费（简单需求跑满 3 轮），要么截断（复杂需求还没澄清完就结束）。让用户决定什么时候够了，比让程序猜更合理。

**为什么用户可以不回答问题？**

用户不一定是领域专家，有时候不知道怎么回答是正常的。强制要求用户回答每个问题会卡住流程。AI 给出合理假设继续推进，最终把假设记录在 Open Questions 里，让 auto-dev 或开发者后续处理——比卡在这里好。

**确认清单有什么用？**

Phase 1 结束后，用户可能已经回答了一些问题，但不确定 AI 有没有理解对。确认清单把"当前 AI 的理解"明确展示出来，让用户在 Phase 2 开始之前发现误解并纠正，而不是等到 Phase 3 生成了一份方向跑偏的文档才发现。

## 快速上手

```bash
# 进入交互式向导
python cli_assistant.py

# 输入需求描述
> 我想做一个...

# 选择模式
[1] 自由讨论
[2] 需求讨论   ← 选这个

# 选择参与的 AI（Claude、Codex、Kimi 等）
# 然后跟着流程走就行了
```

结束后在 `meetings/<议题>/` 目录找到 `requirement.md`，交给 auto-dev 或直接开发。

---

## 适合什么场景

- 想法模糊，不知道从哪里开始定义需求
- 需求文档总是写到一半发现漏了什么
- 想在开发前验证需求是否完整
- 需要把一个想法快速转化成可执行的开发任务

不适合的场景：需求已经非常清楚，只差实现。那直接用 auto-dev 就好了。

---

需求讨论模式解决的问题很简单：**让 AI 帮你把问题问清楚，再开始做事**。这一步通常被跳过，但它往往决定后面的工作是不是在正确的方向上。
